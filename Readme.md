# Система бронирования событий

Разработка системы бронирования для оффлайн и онлайн событий. Проект реализуется на FastAPI с использованием современных подходов к архитектуре и тестированию.

## Описание проекта

Система предоставляет функционал для создания событий, просмотра доступных мест, бронирования и управления пользователями. Проект разделен на две части: сначала разрабатывается прототип API с Pydantic моделями, затем реализуется полноценная логика с подключением базы данных.

## Функциональные требования

### Пользовательская часть
- Регистрация пользователей (самостоятельная)
- Просмотр доступных событий
- Просмотр доступных мест на конкретное событие
- Бронирование мест на события

### Административная часть
- Создание, редактирование и удаление событий
- Управление бронированиями
- Просмотр метрик и аналитики

### Метрики и аналитика
- Рейтинг популярных событий
- Анализ заполняемости мероприятий
- Дополнительные метрики для оценки эффективности системы

## Этапы разработки

### Часть 1: Прототип API
- Разработка Pydantic моделей для всех сущностей
- Создание эндпоинтов без реализации бизнес-логики
- Генерация swagger.json (файл должен быть помещен в корень проекта)

### Часть 2: Полноценная реализация
- Реализация методов согласно принципам SLAP (Single Level of Abstraction Principle)
- Чистая архитектура с разделением на слои
- Подключение базы данных
- Реализация метрик и аналитики
- Покрытие кода тестами

## Технологический стек

- **Python 3.11**
- **FastAPI** — фреймворк для создания API
- **Pydantic** — валидация данных и управление моделями
- **Poetry** — управление зависимостями
- **Pytest** — тестирование

## Запуск проекта

```bash
# Установка зависимостей
poetry install

# Запуск сервера разработки
poetry run uvicorn app.main:app --reload
```

## Документация API

После запуска проекта документация доступна по адресам:
- Swagger UI: http://localhost:8000/docs
- ReDoc: http://localhost:8000/redoc

## Тестирование

```bash
poetry run pytest tests/
```

## Примечания по реализации

- На этапе прототипа бизнес-логика не реализуется, только модели и эндпоинты
- Во второй части код должен соответствовать принципам чистой архитектуры
- Уровни абстракции должны быть четко разделены (SLAP)
- Если для генерации данных используются ML-алгоритмы, это должно быть указано в README
